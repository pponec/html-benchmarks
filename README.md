# HTML Builder Benchmark

This project compares the performance and memory efficiency of several Java HTML generation frameworks and template engines.

## Environment
* **Java Version:** 25.0.2 (java-25-amazon-corretto)
* **Hardware/Memory:** ~15 GiB total RAM available (tested on a machine with 15 GiB RAM)
* **Operation System:** Ubuntu 24.04.4 LTS

## Tested Frameworks
* **j2html:** `1.6.0`
* **HtmlFlow:** `5.0.3`
* **JTE (Java Template Engine):** `3.1.12`
* **Ujorm3:** `3.0.0-RC2`
* **StringBuilder:** *(Native JDK baseline)*

## Test Scenarios & Metrics
The benchmark is powered by **JMH (Java Microbenchmark Harness)** configured with the `-prof gc` profiler. The scenario dynamically builds an HTML table containing three rows of data. The table columns represent the following metrics:

* **Throughput [ops/s]:** The number of HTML generation operations completed per second. Measured in JMH Throughput mode. **Higher is better.**
* **Allocation [B/op] (Normalized Allocation Rate):** The exact amount of temporary Heap memory allocated in **Bytes per operation**. Measured via JMH GC profiler, this metric indicates the volume of "garbage" generated by the framework. **Why it matters:** Lower allocation means significantly less pressure on the Garbage Collector, resulting in lower CPU usage, fewer latency spikes, and better overall execution speed under high server load. **Lower is better.**
* **JAR Size [kB]:** The total file footprint of the framework, including all of its transitive dependencies. Measured in kilobytes. **Lower is better.**
    * *How it is calculated:* The size is dynamically resolved by the `maven-dependency-plugin`, which isolates the specific framework and pulls down its entire dependency tree into a temporary folder to weigh it without the overhead of the JMH benchmark engine itself.
    * *What is a transitive dependency?* When you add a framework to your project, it often requires other libraries to work (e.g., a logging facade or a parsing utility). Those libraries might require even more libraries. This chain is called a transitive dependency tree. Measuring the total size provides a realistic view of how much bloat the framework actually adds to your final application bundle.

## Benchmark Results

The frameworks are sorted from the most productive (highest throughput and lowest memory allocation) to the weakest in this dynamic generation scenario. `StringBuilder` is included purely as an absolute raw-performance baseline.

| Library | Throughput<br/>[ops/s] | Allocation<br/>[B/op] | JAR Size<br/>[kB] |
|:--------|-----------------------:|----------------------:|------------------:|
| StringBuilder *(baseline)* | 6,678,443 | 1,568 | 0 |
| **UjormElement** | **1,110,856** | **1,616** | **114** |
| J2html | 1,037,662 | 4,840 | 198 |
| HtmlFlow | 705,945 | 5,640 | 52 |
| Jte | 625,866 | 8,880 | 79 |

### Key Takeaways
* **UjormElement** is the clear winner among object-oriented builders. It delivers the highest throughput and consumes only **1,616 B/op** — just 48 bytes more than the raw native `StringBuilder` baseline. This is nearly a **3x smaller memory footprint** compared to `j2html`.
* The incredibly low allocation rate means Ujorm creates virtually zero garbage collection pressure during HTML rendering, making it ideal for high-throughput web servers.
* Generating the DOM dynamically per request causes significant overhead in templating engines like **Jte** and strict builders like **HtmlFlow**, pushing their memory allocation rates much higher.
